User Configuration YAML Documentation

Overview

This YAML file defines user attributes and configurations for system provisioning and access management. It is primarily used for automating user creation, setting permissions, and enabling secure authentication via SSH.

Structure

The YAML file consists of a list of users, each with specific attributes. Below is a breakdown of each key and its purpose.

YAML Structure Breakdown

users:
    - identifier: "server-01"
    username: "testuser"
    password: "securepassword123"  # Optional, can be generated if not provided
    permissions: "sudo"  # Optional user group assignment
    create_user_ssh_key: true  # Whether to generate an SSH key for the user
    category: "production"  # Optional, specify category for host lookup

Explanation of Fields

users

  • A list containing multiple user definitions.
  • Each user entry is defined using - under users.

identifier

  • A unique identifier for the server or user.
  • Typically used to distinguish different user entries.

username

  • Specifies the username that will be created on the system.
  • This is the account name used for logging in.

password (Optional)

  • Defines the user password.
  • If not provided, a password can be automatically generated.
  • Security Note: Storing plaintext passwords in YAML is not recommended. Consider using environment variables or secrets management tools.

permissions (Optional)

  • Assigns a user group or access level.
  • Example: sudo grants administrative privileges to the user.
  • If omitted, the user will have default system permissions.

create_user_ssh_key

  • A boolean (true or false) indicating whether an SSH key should be generated for the user.
  • true: Creates a new SSH key for secure authentication.
  • false: No SSH key will be created.
  • Useful for enabling key-based authentication instead of passwords.

category (Optional)

  • Specifies a category or classification for the user.
  • Example: production, staging, or development.
  • Can be used for filtering or role-based access control.

Example Use Cases

  • Automated user provisioning: This YAML can be used with Ansible, Terraform, or custom scripts to automate user creation.
  • Access control management: Assign different permission levels and authentication methods.
  • Secure authentication: Enforce SSH key-based authentication for improved security.

Best Practices

  1. Use Secrets Management: Avoid storing plaintext passwords; use a secrets manager like Vault, AWS Secrets Manager, or environment variables.
  2. Restrict Sudo Access: Grant sudo permissions only when necessary to minimize security risks.
  3. Enable Key-Based Authentication: Set create_user_ssh_key: true for more secure access.
  4. Categorize Users Properly: Use the category field to differentiate production and non-production users.

Conclusion

This YAML configuration simplifies user provisioning, enforces security policies, and ensures consistency across systems. It is ideal for DevOps workflows and automated server setups.

DynamoDB Table Configuration Documentation

Overview

This YAML configuration defines a DynamoDB table setup for deployment. It specifies table attributes, key schema, billing mode, and tagging information.

Configuration Breakdown

dynamodb_tables

  • A list of DynamoDB tables to be created.
  • Each entry in the list represents a table with the following attributes:

Table Properties

  • name: The name of the DynamoDB table (e.g., TestTable).
  • region: AWS region where the table will be created (e.g., us-east-1).

Attribute Definitions

  • attribute_definitions: Specifies the attributes for the table, including:
    • AttributeName: Name of the attribute (e.g., id).
    • AttributeType: Data type of the attribute (S for String, N for Number, B for Binary).

Key Schema

  • key_schema: Defines how the table's primary key is structured.
    • AttributeName: Name of the primary key attribute (e.g., id).
    • KeyType: Type of key (HASH for partition key, RANGE for sort key if applicable).

Billing Mode

  • billing_mode: Determines how read and write capacity is managed.
    • PAY_PER_REQUEST: On-demand mode where AWS charges per request.
    • PROVISIONED: Fixed read/write capacity mode requiring manual throughput settings.

Provisioned Throughput (Only for PROVISIONED mode)

  • provisioned_throughput: Specifies capacity settings when using PROVISIONED mode.
    • ReadCapacityUnits: Number of read capacity units (e.g., 5).
    • WriteCapacityUnits: Number of write capacity units (e.g., 5).

Tags

  • tags: Metadata labels for resource identification.
    • Key: Tag key (e.g., Environment, Project).
    • Value: Corresponding value (e.g., Test, DevOps-Bot).

Example Use Case

This configuration creates a DynamoDB table named TestTable in the us-east-1 region with an id attribute as the partition key. It uses the PAY_PER_REQUEST billing mode and includes relevant environment and project tags.

CodeBuild Project Configuration Documentation

Overview

This YAML configuration defines an AWS CodeBuild project setup. It specifies the project name, environment settings, source repository, and artifact storage.

Configuration Breakdown

codebuild_projects

  • A list of AWS CodeBuild projects.
  • Each entry represents a project with the following attributes:

Project Properties

  • name: The name of the CodeBuild project (e.g., MyCodeBuildProject).
  • region: AWS region where the project will be created (e.g., us-east-1).

Source Configuration

  • source: Defines the source repository for the build.
    • type: Source provider (e.g., GITHUB).
    • location: Repository URL (e.g., https://github.com/myrepo/myproject.git).

Build Environment

  • environment: Specifies the build environment settings.
    • type: The environment type (e.g., LINUX_CONTAINER).
    • image: The build image to use (e.g., aws/codebuild/standard:5.0).
    • computeType: Defines the compute resources (e.g., BUILD_GENERAL1_SMALL).
    • environmentVariables: List of environment variables for the build.
      • name: Variable name (e.g., ENV_VAR1).
      • value: Corresponding value (e.g., value1).

IAM Role

  • service_role: IAM role ARN assigned to the CodeBuild project (e.g., arn:aws:iam::123456789012:role/CodeBuildServiceRole).

Artifacts

  • artifacts: Defines where the build artifacts will be stored.
    • type: Storage type (e.g., S3).
    • location: S3 bucket name for storing artifacts (e.g., my-codebuild-bucket).

Tags

  • tags: Metadata labels for resource identification.
    • Key: Tag key (e.g., Environment, Project).
    • Value: Corresponding value (e.g., Development, DevOps-Bot).

Example Use Case

This configuration creates an AWS CodeBuild project named MyCodeBuildProject in the us-east-1 region. It pulls the source code from GitHub, builds it in a Linux container using AWS CodeBuild's standard image, and stores the resulting artifacts in an S3 bucket. The project is tagged for environment tracking.

Configuration Breakdown
