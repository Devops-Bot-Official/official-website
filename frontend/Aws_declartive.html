User Configuration YAML Documentation

Overview

This YAML file defines user attributes and configurations for system provisioning and access management. It is primarily used for automating user creation, setting permissions, and enabling secure authentication via SSH.

Structure

The YAML file consists of a list of users, each with specific attributes. Below is a breakdown of each key and its purpose.

YAML Structure Breakdown

users:
  - identifier: "server-01"
    username: "testuser"
    password: "securepassword123"  # Optional, can be generated if not provided
    permissions: "sudo"  # Optional user group assignment
    create_user_ssh_key: true  # Whether to generate an SSH key for the user
    category: "production"  # Optional, specify category for host lookup

Explanation of Fields

users

  • A list containing multiple user definitions.
  • Each user entry is defined using - under users.

identifier

  • A unique identifier for the server or user.
  • Typically used to distinguish different user entries.

username

  • Specifies the username that will be created on the system.
  • This is the account name used for logging in.

password (Optional)

  • Defines the user password.
  • If not provided, a password can be automatically generated.
  • Security Note: Storing plaintext passwords in YAML is not recommended. Consider using environment variables or secrets management tools.

permissions (Optional)

  • Assigns a user group or access level.
  • Example: sudo grants administrative privileges to the user.
  • If omitted, the user will have default system permissions.

create_user_ssh_key

  • A boolean (true or false) indicating whether an SSH key should be generated for the user.
  • true: Creates a new SSH key for secure authentication.
  • false: No SSH key will be created.
  • Useful for enabling key-based authentication instead of passwords.

category (Optional)

  • Specifies a category or classification for the user.
  • Example: production, staging, or development.
  • Can be used for filtering or role-based access control.

Example Use Cases

  • Automated user provisioning: This YAML can be used with Ansible, Terraform, or custom scripts to automate user creation.
  • Access control management: Assign different permission levels and authentication methods.
  • Secure authentication: Enforce SSH key-based authentication for improved security.

Best Practices

  1. Use Secrets Management: Avoid storing plaintext passwords; use a secrets manager like Vault, AWS Secrets Manager, or environment variables.
  2. Restrict Sudo Access: Grant sudo permissions only when necessary to minimize security risks.
  3. Enable Key-Based Authentication: Set create_user_ssh_key: true for more secure access.
  4. Categorize Users Properly: Use the category field to differentiate production and non-production users.

Conclusion

This YAML configuration simplifies user provisioning, enforces security policies, and ensures consistency across systems. It is ideal for DevOps workflows and automated server setups.

DynamoDB Table Configuration YAML Documentation

Overview

This YAML file defines the configuration for creating and managing Amazon DynamoDB tables. It specifies table properties such as name, region, schema, billing mode, and tagging.

YAML Structure Breakdown

dynamodb_tables:
  - name: "TestTable"
    region: "us-east-1"
    attribute_definitions:
      - AttributeName: "id"
        AttributeType: "S"
    key_schema:
      - AttributeName: "id"
        KeyType: "HASH"
    billing_mode: "PAY_PER_REQUEST"  # Options: PAY_PER_REQUEST or PROVISIONED
    provisioned_throughput:
      ReadCapacityUnits: 5  # Required only if billing_mode is PROVISIONED
      WriteCapacityUnits: 5
    tags:
      - Key: "Environment"
        Value: "Test"
      - Key: "Project"
        Value: "DevOps-Bot"

Explanation of Fields

dynamodb_tables

  • A list containing multiple DynamoDB table definitions.
  • Each table is defined using - under dynamodb_tables.

name

  • Specifies the name of the DynamoDB table.
  • Example: "TestTable".

region

  • Defines the AWS region where the table will be created.
  • Example: "us-east-1".

attribute_definitions

  • Specifies the attributes of the table and their data types.
  • Example:
    • AttributeName: "id" → Defines the attribute name as id.
    • AttributeType: "S" → Specifies the attribute type (S for String, N for Number, B for Binary).

key_schema

  • Defines the primary key structure of the table.
  • Example:
    • AttributeName: "id" → The key attribute.
    • KeyType: "HASH" → Specifies the partition key (HASH key) for DynamoDB.

billing_mode

  • Defines the billing mode for the table.
  • Options:
    • PAY_PER_REQUEST → Charges based on usage (recommended for unpredictable workloads).
    • PROVISIONED → Requires specifying ReadCapacityUnits and WriteCapacityUnits.

provisioned_throughput (Required only if billing_mode is PROVISIONED)

  • Specifies read and write capacity units.
  • Example:
    • ReadCapacityUnits: 5 → Allocates 5 read units.
    • WriteCapacityUnits: 5 → Allocates 5 write units.

tags

  • Adds metadata tags to the DynamoDB table.
  • Example:
    • Key: "Environment", Value: "Test" → Identifies the environment.
    • Key: "Project", Value: "DevOps-Bot" → Associates the table with a project.

Example Use Cases

  • Infrastructure as Code (IaC): Used in Terraform or AWS CloudFormation for automating DynamoDB provisioning.
  • Cost Management: Selecting PAY_PER_REQUEST helps in cost optimization for unpredictable workloads.
  • Security & Organization: Using tags enables tracking and managing AWS resources efficiently.

Best Practices

  1. Choose Billing Mode Wisely: Use PAY_PER_REQUEST for unpredictable workloads and PROVISIONED for stable, high-traffic tables.
  2. Define Keys Properly: Always define an efficient primary key to avoid hot partitions.
  3. Use Tags for Organization: Categorize tables for cost tracking and access control.
  4. Optimize Provisioned Throughput: If using PROVISIONED, monitor and adjust read/write capacity as needed.

Conclusion

This YAML configuration provides a structured approach to defining and deploying DynamoDB tables efficiently. It is particularly useful for DevOps automation and managing cloud infrastructure effectively.

AWS CodeBuild Project Configuration YAML Documentation

Overview

This YAML file defines the configuration for setting up AWS CodeBuild projects. It specifies project properties such as source repository, environment settings, IAM role, build artifacts, and tagging.

YAML Structure Breakdown

codebuild_projects:
  - name: "MyCodeBuildProject"
    region: "us-east-1"
    source:
      type: "GITHUB"
      location: "https://github.com/myrepo/myproject.git"
    environment:
      type: "LINUX_CONTAINER"
      image: "aws/codebuild/standard:5.0"
      computeType: "BUILD_GENERAL1_SMALL"
      environmentVariables:
        - name: "ENV_VAR1"
          value: "value1"
        - name: "ENV_VAR2"
          value: "value2"
    service_role: "arn:aws:iam::123456789012:role/CodeBuildServiceRole"
    artifacts:
      type: "S3"
      location: "my-codebuild-bucket"
    tags:
      - Key: "Environment"
        Value: "Development"
      - Key: "Project"
        Value: "DevOps-Bot"

codebuild_builds:
  - project_name: "MyCodeBuildProject"
    region: "us-east-1"
    source_version: "main"  # Optional: Branch, tag, or commit
    environment_variables:
      - name: "BUILD_ENV"
        value: "production"
      - name: "DEBUG_MODE"
        value: "false"
    execution_id: "execution-12345"

Explanation of Fields

codebuild_projects

  • A list containing multiple AWS CodeBuild project definitions.
  • Each project is defined using - under codebuild_projects.

name

  • Specifies the name of the CodeBuild project.
  • Example: "MyCodeBuildProject".

region

  • Defines the AWS region where the build project will be created.
  • Example: "us-east-1".

source

  • Specifies the source code repository for the build.
  • Example:
    • type: "GITHUB" → Indicates that the source is hosted on GitHub.
    • location: "https://github.com/myrepo/myproject.git" → URL of the repository.

environment

  • Defines the build environment properties.
  • Example:
    • type: "LINUX_CONTAINER" → Uses a Linux-based build container.
    • image: "aws/codebuild/standard:5.0" → Specifies the CodeBuild image version.
    • computeType: "BUILD_GENERAL1_SMALL" → Defines the compute resource type.
    • environmentVariables → List of environment variables available to the build process.

service_role

  • Specifies the AWS IAM role ARN that CodeBuild will assume to run builds.
  • Example: "arn:aws:iam::123456789012:role/CodeBuildServiceRole".

artifacts

  • Defines the output artifacts storage location.
  • Example:
    • type: "S3" → Stores build artifacts in an S3 bucket.
    • location: "my-codebuild-bucket" → Specifies the target S3 bucket.

tags

  • Adds metadata tags to the CodeBuild project.
  • Example:
    • Key: "Environment", Value: "Development" → Identifies the environment.
    • Key: "Project", Value: "DevOps-Bot" → Associates the build with a project.

codebuild_builds

  • A list of build executions for an AWS CodeBuild project.
  • Example fields:
    • project_name: Specifies the project associated with this build execution.
    • region: Defines the AWS region for the build execution.
    • source_version: (Optional) The branch, tag, or commit used for the build.
    • environment_variables: List of key-value pairs passed to the build execution.
    • execution_id: A unique identifier for the build execution.

Example Use Cases

  • Continuous Integration (CI): Automate builds from a GitHub repository with AWS CodeBuild.
  • Infrastructure as Code (IaC): Use this YAML in AWS CloudFormation or Terraform to provision build environments.
  • Environment Management: Define and manage build environments with variables and permissions.

Best Practices

  1. Use IAM Best Practices: Ensure the service_role has the least privileges necessary for the build.
  2. Optimize Compute Resources: Choose an appropriate computeType based on build resource requirements.
  3. Leverage Environment Variables: Use environmentVariables for dynamic configurations.
  4. Secure Artifacts: Store sensitive build outputs in restricted S3 buckets.

Conclusion

This YAML configuration provides a structured approach to defining and managing AWS CodeBuild projects. It is useful for DevOps automation, CI/CD pipelines, and cloud infrastructure management.
